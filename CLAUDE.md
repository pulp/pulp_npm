## About This Project

`pulp-npm` is a [Pulp](https://pulpproject.org/) plugin that adds npm package support to Pulp. It allows mirroring npm packages from a registry, hosting them locally, and serving them via an npm-registry-compatible API. The plugin is managed by [plugin_template](https://github.com/pulp/plugin_template) — many CI/workflow files are auto-generated and should not be edited directly.

Unit tests run without a live Pulp instance:
```bash
pytest -v --pyargs pulp_npm.tests.unit
```

Run a single unit test:
```bash
pytest pulp_npm/tests/unit/test_serializers.py::TestClassName::test_method_name
```

### Functional Tests
Functional tests require a running Pulp instance. In CI, this is a Docker container. The CI runs them as:
```bash
# Parallel tests
pytest -v --timeout=300 --pyargs pulp_npm.tests.functional -m parallel -n 8
# Sequential tests
pytest -v --timeout=300 --pyargs pulp_npm.tests.functional -m 'not parallel'
```

## Architecture

### Pulp Plugin Pattern
This is a Django app registered as a pulpcore plugin via the `pulpcore.plugin` entry point in `pyproject.toml`. It adds npm-specific content types on top of pulpcore's generic infrastructure (models, tasks, REST API, async pipeline).

### Core Models (`pulp_npm/app/models.py`)
- **`Package`** (extends `Content`): The npm content type. Unique on `(name, version, _pulp_domain)`. The `relative_path` is `{name}/-/{name}-{version}.tgz`.
- **`NpmRemote`** (extends `Remote`): Configures a remote npm registry URL with download policy (immediate/on_demand/streamed).
- **`NpmRepository`** (extends `Repository`): Holds `Package` content. `PULL_THROUGH_SUPPORTED = True` enables serving content from a remote without a full sync.
- **`NpmDistribution`** (extends `Distribution`): Serves content and implements `content_handler()` to generate npm-registry-compatible JSON responses (package metadata with tarball URLs).

### Sync Pipeline (`pulp_npm/app/tasks/synchronizing.py`)
The `synchronize()` task uses pulpcore's declarative content framework:
1. `NpmFirstStage` (a `Stage`) fetches the package metadata JSON from the remote URL, recursively resolves dependencies, and emits `DeclarativeContent` objects.
2. `DeclarativeVersion` orchestrates the remaining stages (artifact downloading, content saving, version creation).

When `deferred_download=True` (on_demand/streamed policies), artifacts are not downloaded immediately.

### API Layer (`app/serializers.py`, `app/viewsets.py`)
Standard DRF serializers/viewsets backed by pulpcore's base classes. ViewSets expose CRUD for packages, remotes, repositories, and distributions. The `NpmRepositoryViewSet` adds a `sync` action that dispatches the `synchronize` task asynchronously.

### Domain Support
All models include `_pulp_domain` to support pulpcore's multi-domain isolation. The `NpmDistribution.content_handler()` builds tarball URLs conditionally based on `settings.DOMAIN_ENABLED`.

### Test Fixtures (`pulp_npm/pytest_plugin.py`)
The plugin registers pytest fixtures for use in functional tests: `npm_bindings` (API client), `npm_repository_factory`, `npm_remote_factory`, `npm_distribution_factory`. Tests use `monitor_task` from pulpcore to wait for async operations.
